# T-008 Installer UX Hardening

## Goal

Harden the installer first-use experience with dry-run output, an installed
doctor check, and repeat-install safety that avoids overwriting user-owned
files.

## Background

T-007 proved that Vibecoding Kit can be installed into a clean target project
and run a minimal governance loop. The next risk is first-use trust: users need
to see what the installer will write, verify that an install is healthy, and
rerun the installer without losing local edits.

T-008 keeps the project shell-first. It improves installer diagnostics and
safety without adding a CLI, changing guard semantics, or introducing new
dependencies.

## Required Work

- Add `installer/init.sh --dry-run`.
- Dry-run must report target, project, profiles, CI choices, files that would
  be created, files that would be skipped because content is identical, and
  files that would conflict because the target already has different content.
- Dry-run must not create or modify the target project.
- Add installed `scripts/ai-doctor.sh` from `core/scripts/ai-doctor.sh`.
- Doctor must verify the generated project has core docs, task/plan state, and
  executable governance scripts, then print `DOCTOR_PASS`.
- Change installer writes so repeat install does not overwrite existing files
  with different content.
- Keep first install behavior compatible with profile overlays by generating
  the full payload before merging into the target.
- Add test-kit coverage for dry-run, doctor, and repeat-install conflict
  preservation.
- Update README and release notes with installer verification commands.

## Forbidden Actions

- Do not add a TypeScript, Node, Python, or compiled CLI.
- Do not add a YAML parser.
- Do not introduce network-dependent installer behavior.
- Do not change command guard risk classification semantics.
- Do not change risk report blocking semantics.
- Do not change drift, closeout, adapter block, or policy semantics.
- Do not retag existing releases.
- Do not overwrite user-owned target files with different content.

## Completion Criteria

- `bash installer/init.sh --dry-run ...` prints `DRY_RUN_PASS` and does not
  create or modify the target.
- A generated target project includes executable `scripts/ai-doctor.sh`.
- Running `bash scripts/ai-doctor.sh` inside a generated target prints
  `DOCTOR_PASS`.
- Re-running the installer over a target with a user-modified generated file
  does not overwrite that file and reports an install conflict.
- Existing clean install, profile install, generated project demo, guards, risk
  report, adapter block, and test-kit coverage still pass.
- README documents dry-run and doctor verification commands.
- `docs/releases/v0.7.0.md` documents scope and known limits.
