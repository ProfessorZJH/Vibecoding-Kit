# T-004 Managed Adapter Blocks and Safe Adapter Updates

## Goal

Add managed adapter blocks and a safe update script so Vibecoding Kit can update
generated agent adapter content without overwriting user-owned content around
that block.

## Background

v0.3.0 added workflow documents and an adapter capability matrix for Codex,
Claude Code, Gemini, Cursor, Cline, Roo, Windsurf, GitHub Copilot, and related
AI coding tools. The next problem is safe maintenance: adapter files will evolve
as prompts, workflows, and guard commands change, but generated projects may
also contain local user notes in those same files.

T-004 introduces a marker-based managed block convention and a small shell tool
for validating and replacing only the managed block. This gives future kit
updates a safe primitive without adding a CLI or syncing to global tool
directories.

## Required Work

- Add a documented managed block marker convention:
  `<!-- VIBECODING-KIT:BEGIN -->` and `<!-- VIBECODING-KIT:END -->`.
- Add `core/scripts/adapter-block.sh` with `--check` and `--update` modes.
- Update core and profile adapter entry files to contain exactly one managed
  block.
- Preserve user content before and after the managed block during updates.
- Add tests for valid, missing, duplicate, malformed, and update-preservation
  scenarios.
- Document the behavior in README and a release note.

## Forbidden Actions

- Do not add a CLI.
- Do not sync to global `~/.codex`, `~/.claude`, or other user directories.
- Do not auto-insert a managed block when markers are missing.
- Do not overwrite content outside the managed block.
- Do not add a template engine.
- Do not parse YAML policies.
- Do not change command, drift, secrets, risk, or closeout guard semantics.

## Test Requirements

## Completion Criteria

- `core/scripts/adapter-block.sh` exists and is executable after installation.
- `adapter-block.sh --check <file>` passes only when the file contains exactly
  one well-ordered managed block.
- `adapter-block.sh --update <file> <template>` replaces only the managed block
  content from the template.
- `adapter-block.sh --update` fails when the target or template has missing,
  duplicate, or malformed markers.
- User content before and after the managed block is preserved byte-for-byte.
- Core and profile adapter files contain the managed block markers.
- Installed full-profile target projects contain the markers.
- `scripts/test-kit.sh` covers the managed block behavior and still passes
  existing prompt, workflow, policy, guard, report, demo, and installation
  checks.
- README links to managed adapter block documentation.
- `docs/releases/v0.4.0.md` documents the Managed Adapter Blocks MVP.
